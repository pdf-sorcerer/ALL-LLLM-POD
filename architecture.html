System Architecture

Runtime Architecture

ALL-LLM Pod is a distributed AI system for realtime text, voice, and multi-person persona conversations.

The system runs across three machines connected through Tailscale: an iPhone client, a Windows GPU orchestration server, and an Apple Silicon laptop for local inference. The iPhone handles the user experience. The Windows PC handles speech recognition, voice synthesis, memory, RAG, and routing. The Mac runs Qwen3-30B-A3B locally.

Internal runtime, training-data pipeline, and Discord inference path

Why the System Is Split

The first version ran on a single Windows PC with a 12 GB RTX 4070 Super. It worked, but everything competed for the same GPU memory: speech recognition, local inference, voice synthesis, and runtime control.

Moving inference to Apple Silicon started as a VRAM workaround. Whisper-small had been running on CPU because the GPU could not comfortably support both ASR and local inference. Once the language model moved to the Mac, the Windows machine could focus on audio, routing, memory, and orchestration while the Mac handled Qwen3-30B-A3B in 4-bit quantization.

The immediate gain was richer persona behavior. The larger shift was architectural: bigger context windows, persistent memory, and additional persona systems became feasible once inference and orchestration no longer fought for the same hardware resources.

Runtime Flow

A request enters through the iPhone app or Discord. Voice requests first pass through WhisperX speech recognition. The orchestrator then assembles the persona prompt, recent conversation, retrieved transcript context, and memory before sending the request to the Mac for local inference.

Text responses can return directly to the client. Voice responses continue through F5-TTS on the Windows PC before streaming back as audio. This keeps the product surface simple while the backend coordinates several specialized services behind it.

Training Data as Infrastructure

The personas depend on speaker-labeled podcast transcripts. Those transcripts are not just source material; they are infrastructure. They feed persona prompts, RAG retrieval, memory injection, and voice-reference selection.

That made transcript fidelity a first-order systems problem. Short utterances, cross-talk, dropped audio, and speaker-label drift do not merely make the transcript messier. They change what the personas learn, retrieve, and imitate.

The most important lesson from the training pipeline was that inference quality is only as useful as the data it is grounded in. A better model can still produce worse behavior if the underlying transcript labels are wrong.

Runtime Queueing

Right now, the system processes expensive requests one at a time. A single user can have a smooth experience because speech recognition, language-model inference, and voice synthesis move through the pipeline cleanly.

Single-lane inference and TTS queueing model

Why Queueing Matters

Queueing is the difference between a demo and a multi-user system. Multiple simultaneous users create contention: text and voice requests compete for the same inference lane, while voice requests continue into a second serialized TTS lane.

Scaling therefore becomes a scheduling problem: which requests can wait, which should stream immediately, which should bypass voice synthesis, and which should route to lighter inference paths.

Discord as External Runtime

Discord changes the project from a personal app into a shared system people can interact with together. Users can summon personas into channels, receive text replies, hear synthesized voice responses in voice chat, and eventually trigger panel-style conversations between multiple hosts.

The first Discord version should stay simple: text input, voice output. A user selects a persona, sends a prompt, receives a generated response, and hears the reply spoken back into the channel. Live speech input can come later.

Longer term, the goal is for the personas to operate more independently: gathering stories, discussing topics with one another throughout the week, and eventually generating their own mirror episode of the podcast.
