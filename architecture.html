System Runtime

Runtime Architecture

This page summarizes how I (and others) engage with the chatbot personas, how audio and text move through the system, how performative it is, operational constraints, and things to-do.

Click to enlarge · Runtime topology and infrastructure boundaries

Operational Flow

iOS
SwiftUI app with a FaceTime-style interface. Handles transcript display, streamed TTS playback, and persona animation. The phone doesn't do inference — it's a frontend. Audio and text generation happen on the server.

Windows Orchestration Server
This is where most of the work happens: WhisperX for speech-to-text, F5-TTS for audio generation, websocket routing, memory injection, queue management, and runtime state. Most timing problems originate here.

Apple Silicon Inference (M4 MacBook Pro)
Qwen3 30B runs locally via mlx, kept resident in unified memory with prompt caching enabled. Both the Windows server and the iOS client can talk to it directly. It functions like a dedicated inference box sitting on the local network.

Discord
Same stack, different surface. Discord adds multi-user access without changing anything underneath.

Bottlenecks

Queue pressure
Inference and TTS are serialized. One user at a time works fine. Each additional user adds queue depth and wait time. This is the most obvious constraint on the system.

Timing
The pipeline is ASR → inference → TTS → playback → UI animation. That's a long chain. Small delays at any step compound. Keeping transcript state, audio, and animation synchronized is the main coordination problem, and it lives on the Windows server.

Model residency
Keeping Qwen3 loaded avoids cold-start delays, but it holds memory permanently. The tradeoffs are memory pressure, what happens if the runtime crashes, and how prompt cache invalidation is handled when context changes.

Retrieval

The current retrieval setup leans too heavily on embeddings. It works, but the transcript data is already well-structured — speakers, timestamps, episode boundaries, segment repairs, attribution. That structure isn't being used.

The plan is to move toward SQL-first retrieval and only add semantic search where it actually improves results. The goal isn't a more sophisticated retrieval system. The goal is one that breaks less often.
