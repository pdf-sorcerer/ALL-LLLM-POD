System Architecture · Lifecycle

Architecture

ALL-LLM Pod is built as a three-stage lifecycle: raw podcast transcripts become structured training data; that training data assembles into deployable chatbots; the live system delivers them in real time over voice and text. Each stage is documented below.

1 · Transcripts → Training Data

The source material is 538 episodes of the All-In Podcast. Each episode splits into two parallel tracks: one for text training signal, one for voice references.

TEXT TRACK AUDIO TRACK ────────── ─────────── Whisper transcription Clip extraction Diarization Quality filter Forensic repair Prosody indexing Classification Reference pool Metadata ↓ ↓ persona signal per-persona F5 input

Text-track quality is non-trivial: WhisperX has decoder-level failure modes that drop real utterances, and NeMo's speaker assignment has attribution-level failure modes that misroute them. Both are addressed by a forensic repair layer (dive.html, forensics.html).

Voice references for TTS are extracted in parallel: clips filtered for quality, indexed by prosody, curated into a per-persona reference pool for zero-shot synthesis.

Both tracks feed Stage 2.

2 · Training Data → Chatbots

Each persona is assembled from several layers.

Persona prompts are derived from the corpus work in Stage 1. The system prompt for each host encodes their voice — how they hedge, what they pivot to, how they push back — without inline examples.

Voice references are the curated F5-TTS reference pool from Stage 1. At synthesis time, the system rotates through reference clips per chunk to produce natural prosody variation.

Persona display assets (avatars, names, location, accent color, mouth positioning) are managed as an outfit organizer — interchangeable visual pieces that compose into each persona's presentation in the mobile client.

LoRA adapters for the language model are a TBD layer. V1 shipped with Mistral 7B plus persona-specific LoRA adapters. V2 currently runs Qwen3-30B-A3B with prompt-only persona conditioning. A future Phase 3 will revisit LoRA training on Qwen3.

3 · Chatbots → Inference

The live system spans three machines: an iOS client, a Windows workstation running the orchestrator, and a Mac running the language model. All three communicate over a Tailscale mesh.

ASR runs on Windows via WhisperX large-v3-turbo on CUDA. The iOS client streams audio to a WebSocket; partial transcripts come back continuously, and a final authoritative pass runs when the user stops speaking.

The language model lives on the Mac. The orchestrator assembles a prompt (base + persona + sliding-window memory + retrieved long-term memories), sends it to the Mac via MLX, and streams the response back.

TTS runs back on the Windows GPU. The response is chunked on sentence boundaries; each chunk goes through F5-TTS using a reference clip drawn from the persona's pool. Audio is converted to PCM and streamed to iOS chunk-by-chunk, so playback starts before the full response has finished generating.

Memory operates at two layers. A sliding window keeps the last ten turns of the active session in prompt context. An agentic memory layer indexes promoted turns with vector embeddings and recalls them by similarity. On session end, the conversation is summarized and the summary becomes available to the next session.

Conversation history is persisted to local SQLite as the source of truth, with backup to a multi-tenant Supabase project.

The iOS client is a SwiftUI app with a contact-list metaphor: each persona is a contact you can DM (text) or FaceTime (voice). A status indicator on the DM view reports whether the inference stack is reachable.
